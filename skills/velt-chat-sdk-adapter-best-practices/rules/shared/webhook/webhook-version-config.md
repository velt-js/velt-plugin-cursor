## Webhook Version Configuration

Velt supports two webhook versions with different verification mechanisms. The adapter defaults to v2 (Advanced).

### v2 — Advanced (Default)

Uses Svix-style HMAC-SHA256 signature verification.

- **Secret format:** `whsec_<base64-encoded-key>` (generated by Velt Console)
- **Verification headers:** `webhook-id`, `webhook-timestamp`, `webhook-signature`
- **Timestamp validation:** Rejects events older than 5 minutes (prevents replay attacks)
- **Setup:** Velt Console → Configurations → Webhook Service → Advanced

```env
VELT_WEBHOOK_SECRET="whsec_dGhpcyBpcyBhIHRlc3Qgc2VjcmV0..."
```

No code changes needed — v2 is the default:

```typescript
createVeltAdapter({
  // webhookVersion defaults to "v2"
  botUserId: "my-bot",
  botUserName: "My Bot",
  organizationId: process.env.VELT_ORGANIZATION_ID!,
  resolveUsers,
});
```

### v1 — Basic

Uses a simple `Authorization: Basic <token>` header.

- **Secret format:** Plain string token
- **Verification:** Checks `Authorization` header matches the token
- **Setup:** Velt Console → Configurations → Webhook Service → Basic

```typescript
createVeltAdapter({
  webhookVersion: "v1",
  webhookSecret: process.env.VELT_WEBHOOK_SECRET!,
  botUserId: "my-bot",
  botUserName: "My Bot",
  organizationId: process.env.VELT_ORGANIZATION_ID!,
  resolveUsers,
});
```

### v1 vs v2 Comparison

| Feature | v1 (Basic) | v2 (Advanced) |
|---------|-----------|---------------|
| Verification | Authorization header | HMAC-SHA256 signature |
| Secret format | Plain token | `whsec_<base64>` |
| Replay protection | No | Yes (timestamp check) |
| Default | No | Yes |

### Key Points

- Use v2 unless you have a specific reason for v1 — it's more secure
- The webhook secret is set in Velt Console and must match your `VELT_WEBHOOK_SECRET` env var
- If verification fails, the adapter returns 401 and logs a warning
